# PRODUCT-REQUIREMENTS.md

## 1. Product overview

### Product name
Premium Authority Website Implementation System

### Product type
A reusable, code-first website starter kit and implementation system for building premium authority websites for high-ticket coaches and consultants.

### Product summary
This product is a modular starter kit and implementation system designed for a premium web design agency. It enables fast and scalable development of premium authority websites using reusable components, blocks, layouts, design tokens, page templates, and controlled style variations.

Strategy, copy, wireframes, layout decisions, visual direction, and media assets are defined by the agency team. AI coding agents are used to implement those decisions in code according to the system’s standards for cleanliness, reusability, maintainability, and speed.

### Core promise
Help the agency translate approved creative direction into premium, production-ready websites faster and more consistently without sacrificing quality.

### Product principles
- Premium over trendy
- Distinct over generic
- Reusable over one-off
- Controlled variation over chaos
- Human-directed creativity over AI improvisation
- Fast execution without AI slop
- Clean implementation over clever hacks
- System quality over visual gimmicks

---

## 2. Problem statement

Most websites for coaches and consultants look generic, weak, or inconsistent. They fail to create instant authority, fail to communicate premium positioning, and fail to reflect the strategic and visual intent of the brand.

Most production systems also break in one of two ways:
1. They are too bespoke, which slows delivery and makes reuse difficult.
2. They are too templated, which creates sameness, low perceived value, and design commoditization.

AI-assisted coding can increase speed, but without strong systems and role boundaries it often produces messy code, redundant patterns, weak implementation decisions, and generic-looking output.

This product exists to solve those problems by combining:
- a premium authority-focused implementation system,
- modular reusable sections and layouts,
- controlled design variation,
- clear human-to-AI handoff rules,
- documentation and implementation guardrails,
- and a scalable delivery workflow.

---

## 3. Product definition

This product is not an AI website designer.

This product is a human-directed implementation system in which:
- the agency team defines strategy and creative direction,
- the AI agent accelerates development and implementation,
- and the final output is reviewed and approved by a human.

The product’s job is not to invent websites from scratch.
The product’s job is to convert approved strategic and creative inputs into clean, reusable, scalable production code.

## 5. Human vs AI responsibilities

### Human-owned responsibilities
- Discovery
- Positioning and strategy
- Offer framing
- Messaging and copy
- Information architecture
- Wireframes
- Layout design
- Visual direction
- Art direction
- Media selection and preparation
- Brand interpretation
- Final QA and approval

### AI-agent responsibilities
- Implement approved pages and sections in code
- Translate wireframes and layout decisions into production-ready UI
- Build and reuse components according to the system
- Apply design tokens, spacing rules, and section variants correctly
- Assemble pages from approved content and structural direction
- Assist with refactoring, cleanup, responsiveness, debugging, and documentation
- Preserve maintainability and code quality
- Flag unclear or missing inputs instead of guessing

### Prohibited AI-agent responsibilities
- Inventing strategy
- Inventing messaging or rewriting approved copy without instruction
- Creating design direction without explicit inputs
- Redesigning approved wireframes autonomously
- Improvising visual hierarchy outside the approved layout intent
- Creating net-new sections when existing approved patterns should be used
- Introducing unnecessary abstractions or speculative architecture
- Shipping unreviewed “creative” deviations disguised as implementation

## 8. Non-goals

This product will not:
- generate complete websites from vague prompts,
- replace human strategy or creative direction,
- optimize for drag-and-drop page-builder workflows,
- support low-end mass-market website production,
- prioritize novelty or flashy motion over clarity and perceived quality,
- allow uncontrolled component creation by AI agents,
- aim to be a no-code website builder,
- or support every business type equally in v1.

This system is intentionally opinionated and implementation-focused.

## 13. Required implementation inputs

Before development begins, each project should provide:
- approved sitemap,
- approved page list,
- approved copy deck,
- approved wireframes or layout references,
- approved visual direction or design references,
- approved brand guidelines or token direction,
- approved media assets,
- required forms and integrations,
- SEO requirements,
- content editing requirements,
- and any section-level notes or implementation constraints.

The implementation system must be optimized to convert these inputs into production-ready websites quickly and consistently.

If inputs are missing, unclear, or contradictory, the AI agent must flag the gap instead of guessing.

## 14. Core features

### 14.1 Reusable design system
The product must include:
- primitive design tokens,
- semantic design tokens,
- typography scale,
- spacing system,
- radius, shadow, border, and surface rules,
- responsive layout rules,
- and theme foundations.

### 14.2 Modular component library
The product must include reusable primitives such as:
- buttons,
- containers,
- headings,
- text blocks,
- badges,
- cards,
- forms,
- inputs,
- accordions,
- and media wrappers.

### 14.3 Section library
The product must include reusable website sections such as:
- hero sections,
- trust/logo strips,
- service grids,
- problem-solution sections,
- process sections,
- testimonial sections,
- case study sections,
- FAQ sections,
- blog/content feed sections,
- CTA sections,
- and footer variations.

### 14.4 Variation system
Each core section type must support controlled variants and preset bundles.

Variation must be based on approved implementation dimensions such as:
- layout,
- density,
- visual tone,
- media placement,
- proof emphasis,
- CTA structure,
- and section styling.

The goal is to produce distinct websites without introducing uncontrolled design sprawl.

### 14.5 Page templates
The product must include starter templates for:
- Home
- About
- Services
- Contact
- Blog index
- Blog post
- Landing page / lead magnet page
- Legal pages

### 14.6 Blog/content system
The product must support:
- article publishing,
- blog index and individual article templates,
- authors,
- featured images,
- metadata,
- content categorization,
- and SEO-ready article structure.

Google documents that Article structured data can help search systems better understand article pages and recommends properties such as `author`, `headline`, `datePublished`, and `author.url`. [web:91]

### 14.7 Content editing system
The product must support easy editing of:
- copy,
- images,
- media assets,
- SEO metadata,
- blog posts,
- redirects,
- and reusable page sections.

The editing system must be simple enough for controlled client updates while protecting structural integrity.

### 14.8 Documentation system
The product must include:
- product overview,
- file map,
- editing guide,
- design system guide,
- component and section inventory,
- variant and preset guide,
- deployment guide,
- debugging guide,
- handoff guide,
- and AI-agent rules.

### 14.9 AI-agent guardrails
The AI agent must act as an implementation assistant, not a creative director.

The AI agent must:
- implement from approved source materials,
- preserve the intent of wireframes and layouts,
- reuse existing components before creating new ones,
- follow token and variant systems,
- keep files readable and well-organized,
- avoid redundant abstractions,
- document important implementation decisions,
- and flag unclear requirements instead of guessing.

The AI agent must not:
- invent copy,
- change messaging strategy,
- redesign layouts without instruction,
- create visual styles outside approved direction,
- introduce unnecessary complexity,
- or generate large amounts of unreviewed code that drift from the system.

## 19. Implementation acceptance criteria

A page or feature implementation is considered complete when:
- the built output matches the approved wireframe and layout intent,
- approved copy is implemented accurately,
- approved media assets are placed correctly and optimized,
- all sections use approved components or justified new patterns,
- styling follows token and variant rules,
- responsive behavior matches defined layout behavior,
- interactions and animations match approved direction,
- code passes linting, type checks, and build validation,
- files are organized according to repo conventions,
- and any deviations from source materials are explicitly documented.

Acceptance criteria are meant to define the specific conditions that must be met for work to be considered complete, so this system should treat them as pass/fail requirements rather than vague preferences. [web:146][web:148]

## 21. Assumptions

This product assumes:
- strategy and creative direction are defined by humans before development begins,
- the implementation team works in a code-first environment,
- AI agents perform best with clear inputs, system constraints, and review loops,
- premium outcomes require both design quality and implementation quality,

PRDs should explicitly document assumptions because assumptions affect planning, risk, and decision-making throughout the build. [web:117][web:133]

## 23. Risks

### Key risks
- Too much flexibility creates design inconsistency
- Too little flexibility creates same-looking sites
- AI agents introduce messy or redundant code
- Inputs are incomplete or unclear at implementation time
- Premium positioning becomes superficial styling instead of real strategic clarity
- CMS/editor access introduces structural breakage
- Reusability compromises distinctiveness

### Mitigations
- Use approved presets and variant rules
- Maintain strict component and section inventories
- Enforce AI-agent guardrails
- Require implementation inputs before build begins
- Use documentation and review checklists
- Keep content editing scoped and controlled
- Define a strong Definition of Done
